# Code Style

Optimize for clarity now over flexibility later. Fewer abstractions, fewer dependencies, fewer flags.

- **Write directly against the current API.** No shims, adapters, deprecation wrappers, or re-exports for legacy behavior.
- **Target the modern web baseline.** Modern browser APIs (`fetch`, `URL`, `AbortController`, `crypto.randomUUID()`, etc.) are assumed; no polyfills. Server-side polyfills are fine when Node.js lacks the API.
- **Use relative imports.** No `@`-style aliases.
- **Import from the defining module. No re-exports — ever.** Every consumer imports a symbol from the file that *defines* it, never through a forwarding hop. This bans both forms a re-export takes: `export { x } from "./other"` / `export type { T } from "./other"` *and* the sneakier import-then-export (`import { x } from "./other"; export { x };`). It also bans a `types.ts` (or any file) that only forwards types from a parent/sibling — `types.ts` must *define* types, not re-export them; if a module has no types of its own, it gets no `types.ts`. A re-export is never "harmless convenience": it hides the real owner, defeats grep-for-importers, and turns one rename into two. When you remove one, repoint every consumer to the defining module (split the import if a consumer pulled a real local symbol from the same file). To find them across a repo, grep **both** forms: `grep -rnE 'export (\*|\{[^}]*\}|type \{[^}]*\}) from'` and `grep -rnE '^export \{[^}]+\};?$'` (the second catches import-then-export; filter out lines containing `from`). Removing the re-export then makes the typechecker flag every lingering consumer — let it.
- **Types live in `types.ts`.** Each logical module gets one `types.ts` for its exported types, interfaces, and type-narrowing helpers (brands, asserts, parse fns). Implementation files import from `./types`; they don't export types themselves. Generated/schema types stay in their generated file — don't re-export through `types.ts`. If files outside the folder import a type, move `types.ts` up to the nearest common ancestor (e.g. shared `messaging/types.ts` when both `email/` and `sms/` need it). Exception: types used by exactly one private helper can stay colocated if they're not part of the module's public surface.
- **Constants live in `constants.ts`.** Shared literal values — magic-number replacements, `as const` lookup maps (`DISCOUNT_RATES`), config tables — go in a sibling `constants.ts`, imported as `./constants`, not scattered inline or lumped into `types.ts`. Same discipline as `types.ts`: it *defines* values, never re-exports them, and moves up to the nearest common ancestor when sibling folders share a constant. One wrinkle: an `as const` map that also sources a derived type (`type T = (typeof MAP)[keyof typeof MAP]`) keeps the value and its derivation together in `constants.ts` — the type reads the value, so splitting them would force an import just to re-derive.
- **A file and a folder each earn their existence.** No single-file folders — a directory holding exactly one file should be flattened up to its parent (a folder implies a module boundary; one file isn't one). And no single-function files — a lone helper colocates with its peers, it doesn't get its own file. Grow the unit until it holds a real cluster (sibling implementation, `types.ts`, `constants.ts`, tests), or flatten it. **Exception: harness-mandated layouts** — a container whose single-file shape is *required* by a tool, not chosen for naming, is not a violation. The canonical case is a skill's `<name>/SKILL.md`: the harness discovers skills by that exact folder-plus-`SKILL.md` shape, so the folder *is* the module boundary the tool imposes; a lone `SKILL.md` never needs a sibling to "earn" the directory (add `references/`, scripts, etc. only when the skill genuinely needs them). Same logic for any other tool-fixed path.
- **Fix race conditions at the root.** Don't mask them with `setTimeout`/`nextTick`/`requestAnimationFrame`. Debouncing and throttling are fine.
- **Vendor-neutral container naming.** In identifiers, comments, and docs we control, say "container engine" / `CONTAINER_HOST`, not "Docker" / `DOCKER_*` — the fleet runs Podman. Never rename externally-defined contracts: `DOCKER_HOST`, `/var/run/docker.sock`, and the docker-compat API are what the Supabase CLI and docker SDKs read; those keep their names.
- **Delete the old shape when you change a shape.** Parse only fields the code actually reads; drop branches that handled prior shapes.
- **Drop unused schema fields.** Schemas describe what the code uses today, not hypothetical past or future clients.
- **Edit schemas in place during prototyping; recreate the DB.** Write migration files only after the schema is shipped to a real environment.
- **Ship features behind real architectural seams** (separate endpoints, separate code paths) rather than runtime feature flags.
- **Use git history as the archive.** Delete dead code instead of commenting it out.
- **Prefer deep refactoring over preserving backwards compatibility.**
- **Surface conflicts; don't average them.** When two existing patterns contradict, pick one — the more recent or better-tested — apply it, and flag the other for cleanup. Don't blend them into a hybrid; two half-followed conventions are worse than one enforced one.
- **Read before you write.** Before adding code, read the exports you'll touch, their immediate callers, and the shared utilities already in play. "Looks orthogonal" is how you duplicate a helper or break a caller you didn't see. If you can't tell why something is structured the way it is, ask before changing it.
